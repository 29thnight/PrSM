# PrSM v2.1 확장 설계

## 1. 목표

v2.0이 "기능 완성과 릴리스 가능 상태"라면, v2.1의 목표는 PrSM을 외부 개발자가 자력으로 학습하고 사용할 수 있는 수준으로 끌어올리는 것이다.

v2.1의 핵심 목표는 세 가지다.

1. Lua 5.5 매뉴얼 / Rust Book 수준의 언어 레퍼런스 매뉴얼 구축
2. 도구 체인(HIR, 소스맵, LSP)의 내부 품질을 실사용 수준으로 향상
3. 첫 공개 릴리스(v0.1.0) 안정화 및 설치 경험 검증

## 2. 비목표

v2.1에서도 다음 항목은 범위 밖으로 둔다.

- 새 언어 기능 추가 (v3 범위)
- 디버거 통합 (VS Code DAP 어댑터 등)
- macOS/Linux 인스톨러 (MSI + winget으로 Windows 우선)
- Unity PlayMode 자동화 테스트 확대

## 3. 축 1: 문서 — 언어 레퍼런스 매뉴얼

### 3.0 현재 상태

현재 문서는 약 635줄(en 기준)이며, 대부분 outline 또는 shallow 수준이다. v2 기능(패턴 바인딩, listen 수명, Input System sugar, 제네릭 추론)은 문서화되지 않았다.

### 3.1 작성 원칙

모든 문서에 적용한다.

1. **코드 예시 필수**: 모든 문법 규칙에 `.prsm` 소스 + 생성 C# 쌍을 보여준다.
2. **에러 예시**: 잘못된 코드와 그 에러 메시지, 수정 방법을 함께 보여준다.
3. **한/영 동시 작성**: `docs/en/`과 `docs/ko/`를 동기화한다.
4. **v2 기능 표시**: v2.0에서 추가된 기능에는 `> Added in v2.0` 배지를 붙인다.
5. **정밀한 의미론**: "이렇게 쓴다" 수준이 아니라, "이것이 어떤 조건에서 어떤 C#로 변환되고, 어떤 경우에 에러인지"를 명시한다.

## 4. 축 2: 도구 체인 품질

### 4.1 HIR 보강

현재 HIR은 심볼 ID + 스팬 수준이다. 다음을 추가한다:

- 해석된 식 타입 (`resolved_type: PrismType`)
- 제네릭 치환 결과 (`generic_substitutions: Vec<(String, PrismType)>`)
- 패턴 구조 상세 (바인딩 타입, 대상 필드/payload 매핑)

효과: LSP hover에서 추론된 타입 표시, code action에서 정확한 리팩토링.

### 4.2 소스맵 v2 sugar 매핑

현재 소스맵은 멤버/문장 수준만 매핑한다. v2 sugar의 세부 매핑을 추가한다:

- 패턴 바인딩 디슈거링 (Item1, Item2 접근 → 원본 바인딩 변수)
- listen 수명 정리 코드 (합성된 `__prsm_cleanup_*` → 원본 listen 문)
- Input System 필드 (`_prsmInput` → 원본 `input.action(...)` 호출)

효과: 디버거에서 생성 코드를 stepping할 때 원본 `.prsm` 위치로 정확하게 매핑.

### 4.3 VS Code C# bridge 소스맵 전환

`csharp-bridge.ts`가 현재 `class ClassName` 패턴 검색으로 C# 위치를 추정한다. `.prsmmap.json`의 앵커 데이터를 사용하여 정확한 위치를 제공하도록 전환한다.

### 4.4 LSP 증분 인덱싱

현재 LSP는 매 요청마다 `build_project_index()`를 호출하여 전체 소스를 재파싱한다. 변경된 파일만 재인덱싱하는 증분 모델로 전환한다:

- `open_documents` 변경 추적 (dirty flag)
- 파일별 `FileSummary` 캐시
- 변경 시 해당 파일만 재파싱 + 심볼 인덱스 갱신

## 5. 축 3: 릴리스 안정화

### 5.1 v0.1.0 첫 릴리스

`git tag v0.1.0 && git push origin v0.1.0`으로 CI 파이프라인 첫 실행. 검증:

- 3개 플랫폼(Windows/macOS/Linux) 바이너리 빌드 성공
- VSIX 패키징 + Marketplace 배포 성공
- MSI 생성 + 설치 E2E (prism --version, VS Code 확장 확인)
- winget-pkgs PR 생성

### 5.2 설치 경험 검증

클린 Windows 환경에서 MSI 설치 후:

1. `prism --version` → 버전 출력
2. VS Code에서 PrSM 확장 활성 확인
3. `.prsm` 파일 생성 → 구문 강조 + 진단 작동
4. `prism build` → C# 생성 확인

## 7. v2.1 종료 조건

다음 조건을 만족하면 v2.1 완료로 본다.

- 문서 총량이 3,000줄 이상 (en 기준)
- 모든 v2 기능(패턴, listen 수명, Input System, 제네릭 추론)이 코드 예시와 함께 문서화됨
- 에러 카탈로그에 전체 에러 코드(E012~E083)가 수록됨
- EBNF 전체 문법이 파서 구현과 1:1로 대응됨
- v0.1.0 MSI 설치 → 사용까지 E2E 검증 완료
- winget `winget install PrSM.PrSM` 동작 확인

## 8. 버전 전략

### 언어 버전 vs 툴 버전 — 분리 운영

| 축 | 체계 | 용도 | 변경 시점 |
|---|---|---|---|
| **언어 버전** | `X.0` (정수.0) | `.prsmproject`의 `language.version`. 문법/의미론 호환성 게이트 | breaking syntax/semantics 변경 시 |
| **툴 버전** | SemVer `X.Y.Z` | 릴리스, 설치, 배포. `prism version` 출력 | 모든 릴리스마다 |

언어 버전은 느리게 올라간다 (1.0 → 2.0 → 3.0). 툴 버전은 릴리스마다 올라간다 (0.1.0 → 0.2.0 → 1.0.0). 하나의 툴 버전이 여러 언어 버전을 지원할 수 있다 (예: 툴 0.2.0이 언어 1.0과 2.0을 모두 지원).

### 컴포넌트 버전 — 모노 (전부 동일)

모든 컴포넌트가 동일한 툴 버전을 사용한다.

| 컴포넌트 | 버전 위치 |
|---------|----------|
| 컴파일러 | `crates/refraction/Cargo.toml` line 3 |
| VS Code 확장 | `vscode-prsm/package.json` line 5 |
| Unity 패키지 | `unity-package/package.json` line 3 |
| MSI 인스톨러 | `crates/refraction/wix/main.wxs` line 21 |

릴리스 시 4곳을 한번에 업데이트한다. `scripts/bump-version.sh`로 자동화.

### Git 태그 규칙

- 릴리스 태그: `v{MAJOR}.{MINOR}.{PATCH}` (예: `v0.2.0`)
- 태그 push 시 GitHub Actions release 워크플로 자동 실행
